## Overview

QtRecon includes a dedicated credential management system for storing and using credentials discovered during penetration tests. Credentials can be associated with specific hosts or domains and automatically injected into tool arguments.

## Credential types

QtRecon supports two primary credential types:

### Password credentials

Plaintext passwords obtained through various means:

- Configuration files
- Database dumps
- Cleartext protocols (FTP, Telnet, HTTP)
- Cracked hashes

### Hash credentials

NTLM or other password hashes for pass-the-hash attacks:

- SAM database dumps
- LSA secrets
- secretsdump output
- Kerberos tickets

## Adding credentials manually

<Steps>
  <Step title="Navigate to credentials tab">
    Click the **Credentials** tab in the main workspace
  </Step>

  <Step title="Create new credential entry">
    Right-click in the credentials table and select **Add credential**
  </Step>

  <Step title="Fill in credential details">
    Enter the following information:
    
    - **Host**: Select the associated host from your project
    - **Type**: Choose `password` or `hash`
    - **Domain**: Domain name or workgroup (can be `localhost` for local accounts)
    - **Username**: Account username
    - **Password/Hash**: The credential value
  </Step>

  <Step title="Save the credential">
    Press Enter or click outside the field to save
  </Step>
</Steps>

## Credential storage schema

Credentials are stored in the `hosts_creds` database table with the following structure:

```sql
CREATE TABLE hosts_creds (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    host_id INTEGER,
    type TEXT,
    domain TEXT,
    username TEXT,
    password TEXT,
    FOREIGN KEY(host_id) REFERENCES hosts(id)
)
```

<Note>
The `password` column stores both passwords and hashes. The `type` field determines how the value should be interpreted.
</Note>

## Importing credentials from files

QtRecon can parse credentials from common formats:

### Secretsdump format

Import directly from Impacket's secretsdump output:

```python
# From core/models/credsmodel.py:87-100
def parse_creds_from_secretsdump(self, output: str) -> list:
    creds = []
    for line in output:
        # SAM hashes: username:uid:rid:lmhash:nthash:::
        if re.match('^[^:]+:[0-9]+:[a-fA-F0-9]{32}:[a-fA-F0-9]{32}:::$', line):
            creds.append({
                'type': 'hash', 
                'username': line.split(':')[0], 
                'value': line.split(':')[3]  # NT hash
            })
        # LSA Secrets: username:lmhash:nthash:::
        elif re.match('^[^:]+:[a-fA-F0-9]{32}:[a-fA-F0-9]{32}:::$', line):
            creds.append({
                'type': 'hash',
                'username': line.split(':')[0],
                'value': line.split(':')[2]  # NT hash
            })
    return creds
```

### User:password format

Simple colon-separated username and password pairs:

```python
# From core/models/credsmodel.py:102-109
def parse_creds_from_user_password(self, output: str) -> list:
    return [{
        'type': 'password',
        'username': line.split(':')[0],
        'value': line.removeprefix(line.split(':')[0] + ':').strip()
    } for line in output]
```

## Using credentials with variables

QtRecon automatically offers to use stored credentials when launching tools that require authentication.

### Credential variables

Tools can reference credentials using these variables:

| Variable | Description | Example |
|----------|-------------|----------|
| `%%%DOMAIN%%%` | Domain or workgroup name | `CORP`, `WORKGROUP` |
| `%%%USERNAME%%%` | Account username | `administrator`, `admin` |
| `%%%PASSWORD%%%` | Plaintext password | `P@ssw0rd123` |
| `%%%HASH%%%` | Password hash (NTLM) | `8846f7eaee8fb117ad06bdd830b7586c` |

### Tool configuration with credentials

Example tools that use credential variables:

## Automatic credential detection

When launching a tool, QtRecon checks if credential variables are present in the command arguments:

```python
# From core/controller.py:442-449
creds_from_database = []
creds_types = [row['type'] for row in Database.request("SELECT DISTINCT type FROM hosts_creds").fetchall()]

for creds_type in creds_types:
    if f"%%%{creds_type.upper()}%%%" in ' '.join(program['args']):
        creds_from_database += Database.request(
            "SELECT DISTINCT type, domain, username, password FROM hosts_creds WHERE host_id = ? AND type = ?",
            (host_dst['id'], creds_type)
        ).fetchall()

if creds_from_database:
    reply = QMessageBox.question(self.ui, 'Valid credentials are available',
                                 f"Valid credentials are available to use against this target. Do you want to use them?")
    if reply == QMessageBox.Yes:
        credtype, domain, username, password = self.view.select_credentials_dialog(creds_from_database)
```

<Note>
QtRecon only prompts for credentials if the tool's arguments contain credential variables AND matching credentials exist for the target host.
</Note>

## Credential scope

Credentials can have different scopes:

### Local credentials

Bound to a specific host, typically for:
- Local administrator accounts
- Service accounts specific to that system
- Credentials extracted from that particular host

### Domain credentials

Valid across multiple systems in the same domain:

```python
# From core/controller.py:445
# Query includes credentials where domain doesn't match localhost or hostname
creds_from_database = Database.request(
    "SELECT type, domain, username, password FROM hosts_creds WHERE "
    "(host_id = ? AND type = ?) OR "
    "(hosts.id = hosts_creds.host_id AND (lower(domain) != 'localhost' and lower(domain) != lower(hosts.hostname)) AND type = ?)",
    (host_dst['id'], creds_type, creds_type)
).fetchall()
```

<Tip>
Set the domain field to the actual domain name (e.g., `CORP`, `EXAMPLE`) for credentials that should be available across all hosts in that domain.
</Tip>

## Credential filtering and search

The credentials table supports filtering to quickly find specific credentials:

```python
# From core/models/credsmodel.py:20-23
if self.filter:
    self.creds = Database.request(
        "SELECT hosts.id as host_id, hosts_creds.id, hosts.ip, hosts.hostname, "
        "hosts_creds.type, hosts_creds.domain, hosts_creds.username, hosts_creds.password "
        "FROM hosts, hosts_creds WHERE hosts.id == hosts_creds.host_id AND "
        "(hosts.hostname LIKE '%' || ? || '%' OR "
        " hosts_creds.domain LIKE '%' || ? || '%' OR "
        " hosts_creds.username LIKE '%' || ? || '%' OR "
        " hosts_creds.password LIKE '%' || ? || '%')",
        (self.filter, self.filter, self.filter, self.filter)
    ).fetchall()
```

Search filters apply to:
- Hostname
- Domain
- Username
- Password/hash value

## Updating credentials

You can modify credential fields directly in the table:

```python
# From core/models/credsmodel.py:83-85
def update_credentials(self, cred_id: str, column: str, new_value: str):
    Database.request(f"UPDATE hosts_creds SET {column} = ? WHERE id = ?", 
                     (new_value, cred_id))
    self.update_data()
```

## Removing credentials

Credentials can be removed individually or in bulk:

```python
# From core/models/credsmodel.py:79-81
def remove_creds(self, creds_id: int) -> None:
    Database.request("DELETE FROM hosts_creds WHERE id = ?", (creds_id,))
    self.update_data()
```

<Warning>
Removing a host from your project automatically deletes all associated credentials. This action cannot be undone.
</Warning>

## Best practices

<Accordion title="Use meaningful domain values">

Set the domain field appropriately:
- `localhost` - Local account specific to this host
- `WORKGROUP` - Default workgroup account
- `CORP` or actual domain name - Domain account valid across multiple systems

This helps QtRecon determine credential scope when prompting.

</Accordion>

<Accordion title="Mark credential types correctly">

Always set the correct type (`password` vs `hash`) to ensure tools receive credentials in the expected format. Some tools only accept hashes, others only accept passwords.

</Accordion>

<Accordion title="Import early in engagement">

If you have credentials from prior knowledge or phishing campaigns, import them early. QtRecon will automatically offer them when launching authenticated enumeration tools.

</Accordion>

<Accordion title="Clean up test credentials">

Remove invalid or test credentials to avoid cluttering the credential selection dialog during tool launches.

</Accordion>
